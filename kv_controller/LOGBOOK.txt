KV Controller Logbook
=====================

This plain-text logbook records what has been implemented so far in the
`kv_controller/` package, what problems came up, what fixes were made, and
what the current results mean.

Project Goal
------------

Build an overlap-aware multi-tier KV page residency controller that:
- operates at the engine / KV-manager layer
- does not modify fused attention kernels
- minimizes decode stall and tail latency when KV spills between CPU and GPU

Thought Process And Failed Selector Ideas
-----------------------------------------

The selector work did not converge in one step. The repo should show that
clearly.

1. First idea: reuse-only trigger

What we tried:
- switch to the reuse/page-stats hybrid whenever average request-local inverse
  reuse was high

Why it looked plausible:
- the reuse-aware scorer should help when pages are being reused soon

What failed:
- both the main benchmark and the hard benchmark show strong request-local
  reuse, so this trigger fired in both regimes
- that made the selector too aggressive on the main benchmark

What we learned:
- reuse exists in both regimes
- reuse alone is not the right discriminator

2. Second idea: reuse plus predicted-page footprint

What we tried:
- switch only when both:
  - request-local reuse was high
  - predicted-page footprint was large

What improved:
- it matched the hard benchmark better

What still failed:
- it was still too eager on the main benchmark
- the few main-benchmark steps with unusually large predicted-page sets were
  enough to give back part of the `normalized` win

What we learned:
- predicted-page footprint matters more than reuse
- but a single hard switch is still brittle

3. Third idea: soft blend between `normalized` and `page_stats_hybrid`

What we tried:
- blend the two scorers based on predicted footprint and working-set size

What failed:
- this preserved the main-benchmark result
- but it threw away too much of the hard-benchmark gain

What we learned:
- the problem is not just "how much page-history signal to add"
- the problem is "which steps should receive that signal at all"

4. Fourth idea: predicted footprint plus working-set threshold

What we tried:
- only switch when:
  - predicted ratio was high
  - required-page count was above a threshold

What improved:
- it protected the main benchmark better

What failed:
- it also removed too much of the hard-benchmark benefit
- hard-regime steps with smaller working sets still mattered

What we learned:
- absolute working-set size is useful context, but too coarse as a hard gate

5. Current best design: prediction-conditioned regime scorer with tuned
   high-pressure weights

What we now use:
- `RegimeAwarePageStatsScorer`
- trigger on large predicted-page footprint
- when triggered, use a tuned page-history scorer with slightly softer weights
  than the plain `PageStatsHybridScorer`

Why this is better:
- it keeps the idea interpretable
- it is grounded in causal step-local signals
- it now slightly improves the hard benchmark beyond the plain page-stats
  hybrid

Current outcome:
- main benchmark:
  - still worse than `normalized`
  - but better than earlier regime-aware attempts
- hard benchmark:
  - now slightly better than `page_stats_hybrid`

So the honest conclusion is:
- the regime-aware direction is real
- the selector is not "100% solved" yet
- but it is now a meaningful novelty path rather than a placeholder idea

To avoid overfitting the selector to only two extremes, we added a middle

  `recent_threshold_round_robin_interleave`
- middle:
  `recent_topk_round_robin_interleave`
- hard:
  `round_robin_interleave`

- `short_reuse_fraction = 0.0815`

Interpretation:
- this is less hostile than the main benchmark
- but still not as reuse-friendly as the dense hard stress benchmark
- it is a useful bridge regime for scorer selection

- the selector problem is now clearly a three-regime problem, not a two-regime

- the current regime-aware scorer can no longer be judged only by:
  - main hostile benchmark
  - hard stress benchmark
- it also needs to behave sensibly on the new middle benchmark

- main benchmark best:

- middle benchmark best:
  - `layer_normalized`
- hard benchmark best:
  - `regime_aware` / tuned page-history scorer

This is a stronger and more novel research story than a single global scorer:
- different scoring strategies are practical in different page-pressure regimes
- the remaining challenge is learning or designing the regime selector cleanly

Step 4: First Adaptive Controller
---------------------------------

What we added:
- `ResidencyController.observe(...)` feedback hook
- simulator callback to `controller.observe(...)`
- `ContextualBanditController` in `kv_controller/policies.py`

How the adaptive controller works:
- it builds a feature vector from the current controller context
- it chooses among a small action set using a LinUCB-style contextual bandit
- it delegates the chosen action to a concrete static controller
- it updates its learned action model after observing outcomes

Original adaptive action space:
- eviction rule: `lru` or `score`
- prefetch depth: `0`, `2`, or `4`

Why Step 4 mattered:
- it introduced adaptive behavior without heavy RL
- it kept the decision space interpretable
- it created the first online learning loop in the project

Important Problems and Fixes
----------------------------

1. Flat stall bug

Problem:
- `stall_ms` was almost identical across all policies, which made the simulator
  look insensitive to policy quality.

Root cause:
- the scheduler only counted the final wait before decode launch
- it did not count time spent waiting in the demand-transfer queue when copy
  lanes were already occupied

Fix:
- in `kv_controller/scheduler.py`, demand-side queue delay is now added to
  `stall_ms` and `transfer_wait_ms`

Why this change was necessary:
- more misses should create more demand-transfer contention
- more contention should increase stall
- without that accounting, the simulator hid real latency differences

2. Bandit cold-start and reward issues

Problem:
- the bandit initially performed worse than the fixed baselines

Challenges:
- the trace was short, so the learner had little time to adapt
- the earlier reward was not emphasizing misses strongly enough
- the environment had just been fixed, so the bandit had previously been
  learning from a weak stall signal

Fixes made:
- lowered exploration
- increased miss penalty
- reduced over-penalization of prefetch and evictions
- added a warm-start bootstrap action
- added previous-step stall, misses, and evictions to the feature vector
- fixed the CLI default `--bandit-alpha` so it matches the tuned controller

Why these changes were made:
- to stabilize the first few learning steps
- to make the reward closer to the project objective
- to avoid judging the bandit from a noisier-than-intended CLI config

3. Added more meaningful adaptive knobs

Problem:
- the original bandit action space only changed eviction rule and prefetch
  depth
- that was not rich enough to express some important engine-level behaviors

New adaptive knobs added:

- `budget_mode`
  - when enabled, the bandit can choose a score-based policy that also emits
    adaptive per-layer HBM budgets
  - the simulator now enforces those budgets in `kv_controller/simulator.py`

- `guard_mode`
  - when enabled, the controller dynamically reduces prefetch depth when the
    system looks stressed
  - the guard looks at things like:
    - transfer backlog
    - very high HBM occupancy
    - combined churn and miss pressure

Why these knobs were added:
- they are more aligned with the actual engine-level control problem
- they give the bandit more meaningful choices than just "prefetch more or less"
- they make the controller more defensible as a scheduling system

4. Bandit credit-assignment fix

Problem:
- prefetch decisions usually help the next step, not the same step
- the original bandit update logic gave too much credit to same-step outcomes

Fix:
- the bandit now uses one-step delayed credit assignment for stall/miss reward
- it still charges the current step immediately for the direct control cost of
  prefetches and evictions

Why this change was made:
- it matches the causal structure of the simulator more closely
- it gives proactive actions a fairer learning signal

New Entry: Step 4 Reward / Replay Round
=======================================

This round focused on three goals:
- improve the bandit's miss/stall tradeoff
- evaluate adaptive behavior across multiple seeds by default
- add an offline trace-replay layer before any live-engine work

1. Richer bandit reward

In `kv_controller/policies.py`, the bandit reward now includes:
- useful prefetch bonus
- wasted prefetch penalty
- miss reduction bonus relative to the previous step
- miss increase penalty relative to the previous step

Why:
- stall alone was not enough
- raw miss penalty alone was not telling us whether prefetch was helping or just
  creating pressure

2. Better bandit diagnostics

In `kv_controller/benchmark.py`, summaries now expose:
- total useful prefetches
- total wasted prefetches
- prefetch hit rate
- total miss reduction
- total miss increase

Why:
- this makes bandit behavior easier to debug and explain

3. Multi-seed policy-suite runs by default

In `scripts/run_kv_controller_sim.py`, policy-suite runs now default to
aggregate evaluation over seeds `0,1,2,3,4` unless a different `--seed-list`
is provided.

Why:
- adaptive-controller conclusions are too noisy on one seed

4. Offline trace replay

Added `kv_controller/replay.py` with helpers to:
- save a trace to JSON
- load a trace from JSON
- replay that trace through the simulator

Why:
- this gives us an offline bridge from synthetic traces to future engine-
  collected traces

This failed with:

What that error means:
- the real trace's required working set was larger than the simulator's current
  HBM capacity
- GPT-2 has 12 layers
- once the sequence crossed 16 tokens with block size 16, each layer needed
  2 blocks/pages
- that means roughly 24 required pages
- the replay evaluator default HBM capacity was only 20 pages

So the simulator was correctly refusing an impossible residency situation.

Why this is still a useful result:
- the real trace collector worked
- the replay file was valid enough to reach the simulator
- the simulator caught a real capacity issue instead of silently continuing

Immediate next implication:
- replaying real traces must use a capacity large enough for the chosen
  model / block size / sequence length

Example next run:

  --trace-json /tmp/real_head_trace.json \
  --hbm-capacity-pages 32 \
  --policies score,bandit

Framework-level vs kernel-level note:

Current collector:
- uses framework-level signals from `transformers`
- captures logical attention tensors with `output_attentions=True`

This is different from kernel-level collection:
- framework-level = higher-level model API outputs
- kernel-level = instrumentation much closer to the fused attention kernel

Why framework-level first:
- simpler
- easier to debug
- enough to validate the head-importance data path before harder integration

What the batch study showed:

1. The batch harness works.
- it collected multiple real traces automatically
- it inferred minimum feasible capacity from each trace
- it replayed those traces under several capacities
- it wrote structured CSV summaries

2. The results were extremely stable across capacities.
- `fox` stayed at:
  - recomputed: miss 22, mean_stall 0.4050, evictions 0
  - layer_normalized: miss 21, mean_stall 0.4050, evictions 0
- `hierarchy` stayed at:
  - recomputed: miss 46, mean_stall 0.5400, evictions 0
  - layer_normalized: miss 44, mean_stall 0.5400, evictions 0
- `memory` stayed at:
  - recomputed: miss 58 or 46 depending on decode configuration, but always
    identical across tested capacities
  - layer_normalized: 1 fewer miss than recomputed, same stall, evictions 0
- `controller` stayed at:
  - recomputed: miss 34, mean_stall 0.2430, evictions 0
  - layer_normalized: miss 33, mean_stall 0.2430, evictions 0

3. Score-based and bandit policies remained identical.
- same misses
- same stall
- same prefetch count
- same zero-eviction behavior

What this means:

The big lesson is more specific than "we need more data."
The current real-trace replay setup is structurally too easy for a different
reason:

- in the real collector, every decode step marks all current causal prefix
  blocks as `required_pages`
- for dense causal attention, that required set grows monotonically as decode
  continues
- the final step's required set is effectively the full set of blocks ever used
- the batch harness then sets replay capacity to `min_required_capacity` or
  slightly above it

So in practice:
- capacity at `min` is already enough to hold the entire lifetime working set
- once pages arrive, they never need to be evicted
- therefore controller choice barely matters

This is why there are:
- no evictions
- almost no prefetch differences
- identical bandit and score behavior
- no sensitivity to capacity changes above `min`

Important interpretation:
- this is not a failure of the bandit
- and not a failure of the scorer
- it is a limitation of the current real-trace semantics

What we have successfully collected already:
- yes, we are collecting per-head information now
- specifically from real model runs we already store:
  - `per_page_head_activity`
  - `query_head_weights`
  - derived `head_weighted_scores`

So the project *has* crossed the threshold of collecting real per-head signals.
What it does *not* yet have is a real-trace formulation that creates realistic
controller pressure.

What the scorer comparison still tells us:
- `layer_normalized` is consistently slightly better than `recomputed` on miss
  count, usually by 1-2 misses
- but because there is still no eviction pressure, that improvement does not
  translate into stall changes
- so scorer ranking on these traces is still weak evidence

Recommended next steps:

1. Change the real-trace formulation, not just the amount of data.
- collecting more traces in the current dense-prefix format will mostly produce
  the same pattern

2. Introduce pressure in one of these ways:
- multi-request replay with interleaved decode steps from several sequences
- trace extraction that distinguishes:
  - mandatory pages
  - highly attended pages
  - lower-priority pages
- top-k attended block replay, where not every historical block is treated as
  equally required
- a local-window-plus-sparse-attended-block trace representation

3. Keep the current collector output because it is still valuable.
- it gives us real per-head signals
- those signals can be transformed into richer replay formats offline

4. The next concrete implementation should probably be:
- an offline converter from dense real traces into a pressure-inducing replay
  trace, for example:
  - always require the most recent block(s)
  - treat the top-k older attended blocks as the scored sparse set
  - let the controller decide residency under a tighter capacity

Conclusion:
- the batch automation succeeded
- real per-head data collection succeeded
- the current bottleneck is replay semantics, not data collection

  8. record accesses
  9. finalize page-wise metrics
- this makes the simulator easier to reason about as a page-wise state machine
  instead of a mixed timing/control routine

- the simulator still uses the same scheduler and state objects

Unified Controller Design And Comparison
----------------------------------------

Why this was necessary:
- the actual end goal is one integrated controller inside a vLLM-like pipeline
- not three separate permanent systems
- so we turned the scorer lessons into a single-controller design track

Design doc:
- `kv_controller/UNIFIED_CONTROLLER_DESIGN.md`

Implemented controller options:
- `UnifiedRuleController`
- `UnifiedBlendController`
- `UnifiedBanditController`

All three use the same controller interface:
- one `decide(context) -> PolicyOutput`
- one page-priority path
- one eviction/prefetch decision point

  - it no longer collapses on the hard benchmark
  - it nearly matches the unified-rule / unified-blend hard-regime win
  - it still does not beat the older `bandit` on the main or middle regimes

Head-scoring status:
- the head-scoring idea is working
- normalized head score is still strong in the hostile and middle regimes
- we do not need a trained predictor yet for head scoring
- current evidence still favors the analytic scorer path

Reuse/training-data status:
- reuse-aware scoring is working in the hard regime
- we do not yet need training data for reuse
- the next bottleneck is controller/selector design, not lack of labels

Adaptive Unified-Controller Update
----------------------------------

What was wrong before:
- the first `UnifiedBanditController` was too static in practice
- it had:
  - weak features
  - no delayed credit assignment
  - no useful warm start
  - too little action diversity
- result: it was too conservative and clearly worse than the rule/blend
  unified controllers

What we changed:
- reused the delayed-credit update idea from the older contextual bandit
- added previous-step features:
  - previous stall
  - previous misses
  - previous evictions
- expanded the action space across:
  - scorer mode
  - prefetch depth
- added a heuristic prior / warm-start action based on current regime signals

Interpretation:
- the adaptive unified controller is now genuinely competitive in the hard

- it still has a generalization problem on the main and middle regimes

Current honest conclusion:
- one unified controller is feasible
- the best current unified designs already win in the hard regime
- the remaining problem is making that same controller preserve the main and
